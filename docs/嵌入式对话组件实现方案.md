# 嵌入式对话组件实现方案

> 状态: 已实施
> 日期: 2026-05-13
> 方案: iframe + postMessage 嵌入
> 认证: 基于 API Key（`sk-xxx`），通过 middleware bcrypt 验证，详见 `docs/统一认证系统实现方案.md`

## 概述

提供一段前端代码，让其他 Web 应用以浮动窗口方式集成 open-chat 对话功能。
交互逻辑：浮动图标（可拖动）→ 点击弹出对话窗口 → 关闭恢复图标。
所有 UI 控件（标题栏、☰、×、拖拽、resize）由外层 `embed.min.js` 管理，iframe 仅渲染纯对话内容。

## 最终架构

```
用户网站 (宿主页面)
  └── embed.min.js (在宿主 DOM 中，纯 vanilla JS)
       ├── [浮动按钮]      ← 宿主 DOM 中渲染，支持拖拽
       └── [窗口容器]      ← 宿主 DOM 中渲染
            ├── [标题栏]   ← ☰ + 标题文字 + ×，颜色由 headerStyle 配置
             ├── [iframe]   ← 加载 /embed?apiKey=...&theme=...
            │    └── /embed 页面 (Next.js，复用 Main 组件的布局)
            └── [resize]   ← 右下角拖拽 handle，min/max 约束

控件归属:
  ☰ → 外层，通过 postMessage 发 toggle-sidebar 到 iframe
  × → 外层，直接 closeWindow()
  拖拽 → 外层标题栏
  resize → 外层右下角
  主题色 → 外层容器背景 + 标题栏颜色（headerStyle）
  iframe 内主题 → 通过 theme URL 参数传入 CSS 变量
```

## openChatConfig 配置接口

```javascript
window.openChatConfig = {
  // 必填
  baseUrl: 'http://localhost:3000',   // open-chat 服务器地址
  apiKey: 'sk-xxx',                   // API Key（由管理面板生成，匹配 api_keys 表）

  // 可选
  agentId: 'xxx',                     // 预设智能体
  icon: 'robot',                      // 内置图标名（14 个可选），默认 'robot'
  iconUrl: 'https://...',             // 自定义图标 URL（优先级高于 icon）
  windowTitle: 'AI 助手',             // 窗口标题栏文字
  theme: 'light',                     // light | dark | tech-blue
  locale: 'zh-Hans',                  // 语言

  // 窗口尺寸
  windowSize: {
    width: 400,          // 初始宽度(px)，默认 400
    height: 600,         // 初始高度(px)，默认 600
    minWidth: 320,       // 最小宽度，默认 320
    minHeight: 400,      // 最小高度，默认 400
    maxWidth: 800,       // 最大宽度，默认 800
    maxHeight: 900,      // 最大高度，默认 900
  },

  // 标题栏样式
  headerStyle: {
    backgroundColor: '#1C64F2',         // 标题栏背景，不填按主题取默认值
    borderColor: 'rgba(255,255,255,0.2)',  // 标题栏底边颜色
    textColor: '#ffffff',              // 标题文字颜色
    iconColor: '#ffffff',              // ☰ × 图标颜色
    buttonHoverBg: 'rgba(255,255,255,0.15)',  // 按钮悬停背景
  },

  // 浮动按钮样式
  bubbleStyle: {
    backgroundColor: '#1C64F2',        // 按钮背景色（无 iconUrl 时）
  },

  // 预设参数（填入智能体的 prompt_variables）
  inputs: { key: 'value' },
}
```

## 数据模型

### app_integrations 表（替代旧 embed_tokens）

```sql
CREATE TABLE IF NOT EXISTS app_integrations (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  description TEXT DEFAULT '',
  app_id TEXT NOT NULL UNIQUE,
  app_secret TEXT NOT NULL DEFAULT '',
  is_enabled INTEGER NOT NULL DEFAULT 1,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
)
```

### api_keys 表

```sql
CREATE TABLE IF NOT EXISTS api_keys (
  id TEXT PRIMARY KEY,
  integration_id TEXT NOT NULL,
  name TEXT NOT NULL DEFAULT '',
  key_prefix TEXT NOT NULL,
  key_hash TEXT NOT NULL,           -- bcrypt hash，非明文
  allowed_agent_ids TEXT NOT NULL DEFAULT '[]',
  expires_at INTEGER,
  last_used_at INTEGER,
  is_enabled INTEGER NOT NULL DEFAULT 1,
  created_at INTEGER NOT NULL,
  FOREIGN KEY (integration_id) REFERENCES app_integrations(id) ON DELETE CASCADE
)
```

### TypeScript 类型

```typescript
// types/auth.ts
interface AppIntegrationRecord {
  id: string
  name: string
  description: string
  app_id: string

  updated_at: number

interface ApiKeyRecord {
  id: string
  integration_id: string

  key_prefix: string
  key_hash: string               // bcrypt hash
  allowed_agent_ids: string[]
  expires_at: number | null
  last_used_at: number | null
  is_enabled: boolean
  created_at: number
}
```

## API Key 校验流程

```
API 请求到达 middleware
  ↓
读取 x-api-key header (sk-xxx)
  ↓ 无 → 检查其他认证方式 (JWT cookie, Authorization header)
  ↓ 有
bcrypt 比对 api_keys.key_hash
  ↓ 不匹配 → 401 Invalid API key
  ↓ 匹配
检查 is_enabled === true
  ↓ false → 403 API key disabled
  ↓ true
检查 expires_at (null = 永不过期)
  ↓ 已过期 → 401 API key expired
  ↓ 未过期
检查 agentId in allowed_agent_ids (或 '*')
  ↓ 否 → 403 Agent not allowed
  ↓ 是 → 放行，注入 x-auth-integration-id 到请求头
```

## 现有模式兼容性

三条防线确保零影响：
1. **数据层** — `CREATE TABLE IF NOT EXISTS`，不修改旧表结构
2. **认证层** — middleware 检查 `AUTH_ENABLED`，关闭时放行所有请求
3. **组件层** — `isEmbed` 默认 `false`，渲染路径与现在完全一致

## basePath 前缀配置

项目支持通过 `NEXT_PUBLIC_BASE_PATH` 环境变量配置路由前缀（默认为空，即根路径）：

- `next.config.js` 读取该变量设置 Next.js `basePath`，自动处理页面路由和静态资源前缀
- `config/index.ts` 导出 `BASE_PATH` 常量，客户端 API 调用统一使用 `${BASE_PATH}/api/...` 格式

使用示例：
```bash
# .env.local
NEXT_PUBLIC_BASE_PATH=/chat
```
设置后，所有页面路由变为 `/chat/`、`/chat/embed`，API 变为 `/chat/api/...`。

对于嵌入组件集成方，`baseUrl` 和脚本 `src` 需包含前缀：
```html
window.openChatConfig = {
  baseUrl: 'http://your-server.com/chat',
}
<script src="http://your-server.com/chat/embed.min.js" defer></script>
```
