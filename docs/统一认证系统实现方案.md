# 统一认证系统实现方案

## 概述

为 open-chat 系统添加全局 token 验证。支持两种认证级别：

- **Level 1 (API Key)**: 静态 `sk-xxx` 密钥，用于简单嵌入集成
- **Level 2 (OAuth)**: 服务端到服务端 `app_id` + `app_secret` 交换 JWT，携带用户身份

两种级别共享同一数据库 Schema 和中间件。通过 `AUTH_ENABLED` 环境变量控制是否启用认证（默认关闭，向后兼容）。

## 认证流程

### Level 1: 静态 API Key

```
第三方管理员:
  1. 在管理面板创建集成
  2. 创建 API Key (sk-xxx)
  3. 将 key 配置到 openChatConfig.apiKey

运行时:
  embed.min.js → iframe → API 请求携带 x-api-key: sk-xxx
  → 中间件验证 key，无用户身份
```

### Level 2: OAuth Client Credentials

```
第三方:
  1. 在管理面板注册应用 → 获得 app_id + app_secret
  2. 拥有自己的用户登录系统

后端流程:
  a. 第三方后端认证用户（自有系统）
  b. 后端调用我们的 /api/auth/exchange 接口:
       POST /api/auth/exchange
       { app_id, app_secret, user_id, user_email?, user_name? }

  c. 服务端:
       - 验证 app_id + app_secret
       - 创建/更新用户记录
       - 签发 JWT: { sub: user_id, type: 'user', app_id, exp: 8h }
       - 返回: { access_token, token_type: 'Bearer', expires_in: 28800 }

  d. 第三方后端将 JWT 传给前端

  e. Widget 使用 JWT:
       Authorization: Bearer eyJhbGci...
       → 中间件验证 JWT 签名 + 过期时间
       → 注入 x-auth-user-id 到请求头
```

## 数据库 Schema

### users 表

```sql
CREATE TABLE IF NOT EXISTS users (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL DEFAULT '',
  role TEXT NOT NULL DEFAULT 'user',
  org_id TEXT,
  is_enabled INTEGER NOT NULL DEFAULT 1,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
)
```

### user_accounts 表

```sql
CREATE TABLE IF NOT EXISTS user_accounts (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  login_type TEXT NOT NULL,
  login_identifier TEXT NOT NULL UNIQUE,
  password_hash TEXT NOT NULL DEFAULT '',
  is_primary INTEGER NOT NULL DEFAULT 0,
  is_verified INTEGER NOT NULL DEFAULT 0,
  created_at INTEGER NOT NULL,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
)
```

设计要点：
- `login_identifier` 全局唯一（跨所有 login_type）
- 支持 username、email、phone、wechat 等类型
- 用户可绑定多种登录方式，所有方式指向同一 user_id
- `password_hash` 为空表示非密码登录方式

### app_integrations 表（替代 embed_tokens）

```sql
CREATE TABLE IF NOT EXISTS app_integrations (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  description TEXT DEFAULT '',
  app_id TEXT NOT NULL UNIQUE,
  app_secret TEXT NOT NULL DEFAULT '',
  is_enabled INTEGER NOT NULL DEFAULT 1,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
)
```

### api_keys 表

```sql
CREATE TABLE IF NOT EXISTS api_keys (
  id TEXT PRIMARY KEY,
  integration_id TEXT NOT NULL,
  name TEXT NOT NULL DEFAULT '',
  key_prefix TEXT NOT NULL,
  key_hash TEXT NOT NULL,
  allowed_agent_ids TEXT NOT NULL DEFAULT '[]',
  expires_at INTEGER,
  last_used_at INTEGER,
  is_enabled INTEGER NOT NULL DEFAULT 1,
  created_at INTEGER NOT NULL,
  FOREIGN KEY (integration_id) REFERENCES app_integrations(id) ON DELETE CASCADE
)
```

## JWT 结构

```json
{
  "sub": "user-uuid",
  "type": "user",
  "role": "admin",
  "app_id": "chat-widget-prod",
  "iat": 1715678400,
  "exp": 1715707200
}
```

- 系统用户登录：session cookie（无 maxAge，关闭浏览器后自动过期）
- Level 2 OAuth 交换：有效期 8 小时

## 中间件 Token 解析顺序

1. Cookie `auth_token` → 验证 JWT → 注入 `x-auth-user-id`, `x-auth-user-type`, `x-auth-user-role`
2. Header `x-api-key` 或 `Authorization: Bearer sk-xxx` → 查找 `api_keys` → 检查 `is_enabled`, `expires_at`, `allowed_agent_ids` → 注入 `x-auth-integration-id`, `x-auth-key-id`
3. 无凭证 → API 路由返回 401；页面请求重定向 `/login`

公开路径（跳过认证）：`/api/auth/login`, `/api/auth/logout`, `/api/auth/setup`, `/api/auth/exchange`, `/login`, `/embed`, `/_next`, `/favicon`, `/images`

注意：`/setup` 页面通过 Server Component 做用户存在性检查（有用户则 `redirect('/login')`），不依赖 middleware。`/images` 为嵌入图标等静态资源。

## API 路由

| 路由 | 方法 | 认证 | 说明 |
|------|------|------|------|
| `/api/auth/setup` | POST | 无 | 首次运行管理员创建（自动清理孤立 user_accounts） |
| `/api/auth/login` | POST | 无 | 系统用户登录 → JWT cookie |
| `/api/auth/logout` | POST | JWT | 清除会话 |
| `/api/auth/me` | GET | JWT 或 API Key | 当前用户信息（API Key 认证返回默认用户） |
| `/api/auth/exchange` | POST | app_id + app_secret | Level 2: 交换凭证 → JWT |
| `/api/admin/users` | CRUD | JWT (admin) | 用户管理 |
| `/api/admin/integrations` | CRUD | JWT (admin) | 集成管理 |
| `/api/admin/integrations/[id]/keys` | CRUD | JWT (admin) | API Key 管理 |

## 嵌入认证流程

### 认证链路

```
embed.min.js (外层)
  → iframe /embed?apiKey=sk-xxx
    → Main 组件读取 apiKey prop
      → init: fetch('/api/parameters', { headers: { 'x-api-key': apiKey } })
      → init: fetch('/api/config/agents', { headers: { 'x-api-key': apiKey } })
      → init: RemoteStorageProvider.setApiKey(apiKey)
        → 所有 storage fetch 自动携带 x-api-key
      → AgentSelector: fetch('/api/config/agents', { headers: { 'x-api-key': apiKey } })
      → sendChatMessage: body.apiKey → service 传递 x-api-key header
```

### middleware 处理

```
请求到达 → 提取 x-api-key header
  → 格式检查 (sk- 前缀)
  → 遍历 api_keys 表，bcrypt.compare 验证
  → 通过 → 注入 x-auth-integration-id, x-auth-key-id
  → 路由处理器读取这些 header 确认认证状态
```

### 关键注意

- API Key 必须通过管理面板创建（bcrypt hash 存储），明文 key 无法通过验证
- `RemoteStorageProvider` 需通过 `setApiKey()` 注入 key，否则 storage API 返回 401
- `/api/auth/me` 兼容 API Key 认证：检查 `x-auth-integration-id`，返回默认用户
- 嵌入图标（`/images/embed-icons/`）需在 `PUBLIC_PATHS` 中，否则被 middleware 拦截

## 嵌入系统变更

### openChatConfig 命名

```js
// 旧
window.openChatConfig = { token: 'sk-xxx' }

// 新
window.openChatConfig = { apiKey: 'sk-xxx' }
```

### iframe URL 参数

```
/embed?apiKey=sk-xxx&theme=dark
```

### HTTP Header

```
x-api-key: sk-xxx
```

## 命名规范

| 概念 | 名称 | 使用位置 |
|------|------|---------|
| 集成标识符 | `app_id` | 管理面板、OAuth 交换请求 |
| 集成密钥 | `app_secret` | 仅 OAuth 交换（服务端到服务端） |
| API 认证凭证 | `api_key` | `sk-xxx` 在请求中发送 |
| openChatConfig 字段 | `apiKey` | 客户端嵌入配置 |
| HTTP Header | `x-api-key` 或 `Authorization: Bearer sk-xxx` | 所有 API 请求 |
| 用户登录名 | `login_identifier` | 登录表单、user_accounts 表 |
| 用户显示名 | `name` | UI 显示 |

## 向后兼容

| 场景 | 行为 |
|------|------|
| `AUTH_ENABLED=false`（默认） | 中间件放行所有请求，零影响 |
| 现有 `embed_tokens` 数据 | 首次运行时自动迁移到 `app_integrations` + `api_keys` |
| 旧 `embed.min.js` 配置 (`token: 'xxx'`) | 集成方需更新为 `apiKey` |
